# ML Intern Prompt: Blackline Atlas v2.2 Calibration Dataset

You are an ML data engineer working on a satellite vision-language dataset. You do not need access to the application codebase. Your task is to repair and extend an existing Hugging Face dataset for a civilian, non-tactical satellite disruption triage model.

Use the existing dataset `ChrisRPL/satellite-disruption-triage-aux-v2-1` as the starting point. Do not replace it destructively. If you have write access, add a versioned v2.2 dataset slice or revision to the same dataset repo with clear file names. If the Hub tooling requires a new repo, create `ChrisRPL/satellite-disruption-triage-aux-v2-2` and explain why. Preserve v2.1 provenance.

The purpose of v2.2 is not to make the dataset much larger. The purpose is to create a high-quality calibration and gold-eval pack that fixes the current model bottleneck: the trained VLM under-calls positive disruption cases and predicts zero `downlink_now` on positive smoke tests. Build a clean evidence-first dataset that teaches visible macro-scale civilian disruption cues before the final action label.

Focus only on civilian, macro-scale, human-caused conflict or explosion disruption visible in paired satellite imagery. Good positive evidence includes collapsed or missing building footprints, debris fields, blast or crater scarring, broad urban destruction, damaged civilian warehouses, damaged port or logistics aprons, damaged bridges or access spans, damaged water or power facilities, damaged markets, hospitals, schools, residential blocks, aid hubs, or civilian logistics sites. Do not include military bases, weapons systems, troop positions, convoy intelligence, route-open analysis, targeting support, or tactical recommendations.

Produce 100 to 150 excellent calibration examples with valid paired baseline and current images. The target balance is approximately 40 percent `downlink_now`, 30 percent `defer`, and 30 percent `discard`. Include hard negatives intentionally: unchanged civilian sites near conflict zones, SAR speckle or modality artifacts, low visibility/cloud cases, seasonal or lighting changes, construction or non-conflict change, and near-conflict locations with no visible site damage. Do not use metadata-only synthetic rows in eval or calibration. Avoid natural-disaster examples unless they are clearly marked auxiliary and kept out of the core conflict calibration set.

Each row must use this exact evidence-first schema, with `triage_action` last: `row_id`, `example_id`, `split`, `baseline_image`, `current_image`, `location_name`, `country`, `source_event`, `source_dataset`, `modality`, `baseline_date`, `current_date`, `visual_evidence_tags`, `evidence_strength`, `damage_mechanism`, `visibility_quality`, `negative_type`, `bbox_norm`, `bbox_quality`, `change_confidence`, `civilian_infrastructure_type`, `rationale`, `triage_action`, `provenance`, `license`.

Allowed `triage_action` values are `discard`, `defer`, and `downlink_now`. Allowed `visual_evidence_tags` are `no_visible_change`, `low_visibility`, `sar_speckle_or_modality_artifact`, `seasonal_or_lighting_change`, `construction_or_non_conflict_change`, `collapsed_building`, `roof_loss`, `missing_building_footprint`, `debris_field`, `burn_scar`, `blast_or_crater_scarring`, `damaged_warehouse_block`, `damaged_port_or_logistics_apron`, `damaged_bridge_or_access_span`, `damaged_water_or_power_facility`, `damaged_market_or_civilian_cluster`, `large_rubble_field`, and `broad_urban_destruction`. Allowed `evidence_strength` values are `none`, `weak`, `moderate`, and `strong`. Allowed `damage_mechanism` values are `none`, `explosion_blast`, `airstrike_or_artillery`, `ground_assault`, `fire_burning`, `structural_collapse`, `access_obstruction`, `flood_inundation`, `earthquake_shaking`, `unknown_conflict`, `unclear_human_made`, `non_conflict_change`, `modality_artifact`, and `low_visibility`. Allowed `visibility_quality` values are `excellent`, `good`, `fair`, `poor`, and `unusable`. Allowed `negative_type` values are `none`, `unchanged_civilian_site`, `near_conflict_no_damage`, `low_visibility_cloud`, `sar_speckle_artifact`, `seasonal_lighting_change`, `construction_non_conflict`, and `modality_mismatch`. Allowed `bbox_quality` values are `null`, `tight`, `coarse`, and `weak_whole_tile`. Allowed `civilian_infrastructure_type` values are `none`, `residential_block`, `apartment_complex`, `port_logistics_apron`, `market_bazaar`, `warehouse_storage`, `water_treatment`, `hospital_clinic`, `school_university`, and `bridge_access_span`.

Use strict split hygiene. Create `train_calibration_flat.jsonl`, `eval_gold_flat.jsonl`, `train_calibration_sft.jsonl`, and `eval_gold_sft.jsonl`. The eval gold split must be event-held-out and location-held-out from train. No same city, same event family, or near-duplicate crop should appear in both train and eval. The eval gold split should contain at least 45 examples and should be balanced enough to measure downlink recall, defer calibration, and false positives.

For SFT rows, provide two images and a short user prompt that asks the model to compare baseline and current imagery and return strict JSON. The assistant response must contain only the evidence-first JSON object, with `triage_action` as the final key. Keep rationales short, visual, and non-tactical.

Run validation before upload. Validation must prove that every JSONL row parses, every referenced image exists and is downloadable, every image is a valid PNG or JPEG, all enum values match the allowed taxonomy, all bbox values are normalized with valid order, `downlink_now` rows have a non-null defensible bbox, null bbox rows use `bbox_quality: null`, no duplicate row IDs exist, train and eval have no event or location overlap, and class counts are reported.

Upload the final dataset to Hugging Face with a complete dataset card. The README must clearly explain the civilian scope, non-tactical exclusions, schema, split policy, source datasets, licenses, class balance, known limitations, and the intended use as calibration/eval data for a conflict-disruption VLM. Include `metadata.json`, `validation_report.md`, and `source_audit.md`.

At the end, report the live Hub URL, exact file list, total repo size, total image count, train and eval row counts, class balance per split, source-event coverage, license summary, validation pass/fail table, two verified downloadable image paths, and the three biggest limitations. Do not submit a new fine-tuning job yet. The next training run should happen only after this v2.2 calibration dataset is validated.
